# CLAUDE.md

## Definition of Done

A story is done when ALL of the following are true:

1. **Lint clean** — `npm run lint` exits 0 (Next.js ESLint config).
2. **Type-safe** — `npx tsc --noEmit` exits 0.
3. **Build succeeds** — `npm run build` exits 0.
4. **Tests pass** — `npm run test:e2e` exits 0 (smoke + story-specific tests).
5. **AC coverage** — every acceptance criterion has at least one automated test
   or a documented manual verification step in the story file.
6. **No hardcoded UI strings** — all user-facing text comes from
   `messages/pt-PT.json`; no raw Portuguese (or any language) string literals
   in component JSX. New keys must be added to **every** locale file in
   `messages/` (`pt-PT.json` AND `en.json`) — key parity is enforced by
   `e2e/i18n-key-parity.spec.ts` (added after BUGFIX-01: STORY-17 shipped
   keys only to pt-PT.json and `/en/*` rendered raw key names in production).
7. **No regressions** — previously passing tests still pass.

## Quality gates (CI enforces these)

- `npm run lint`
- `npm run build`
- `npm run test:e2e` (Playwright smoke suite)

## Retry budget

Implementation agents have **2 fix cycles** after first review before the issue
is escalated to a human.

## Complexity classification

- **trivial** — mechanical change, single file, no reasoning risk.
- **standard** — multi-file, requires understanding of at least two modules; default.
- **complex** — auth, data integrity, concurrency, security, or three or more interacting systems.

When in doubt, classify as `standard`.

Reasoning-risk signals that override line-count estimates:
- **Touches inherited/global CSS properties** (`color-scheme`, `color`, `font`, etc.) — blast radius is all descendants app-wide, not apparent from the edit site. Requires full-tree audit to confirm. Upgrade from `trivial` to `standard` even if diff is tiny (BUGFIX-04).
- **Modifies shared state or configuration** (auth guards, routing regexes, env-var contracts) — subtle logic bugs have app-wide scope. Stay `standard` or higher regardless of line count.

## Multi-story delivery (combined PRs)

When two stories are delivered together in a single PR and both touch a shared
file (e.g. two settings-page features editing the same parent page, or two
locale features editing the same translation JSON), assign an explicit
landing order at the orchestrator level (e.g. "story A's implementer lands
first, story B's implementer rebases onto it") rather than leaving ownership
of the shared file ambiguous between two independently-refined plans. Add a
shared re-check step to the second story's plan (e.g. re-verify translation
key parity across both locale files) so integration gaps are caught before
Review, not during it. See CHORE-06/CHORE-11 (PR #27) for a worked example:
the settings page and `messages/*.json` were shared between both stories, and
the Challenge stage flagged the ambiguity before Implementation started.

## Scope firewall for reuse planning (STORY-30 lesson)

When a story plan proposes to reuse a constant or shared export from another
module (e.g., hoisting `SUNDAY_HORIZON` from `lib/availability/upcoming-sundays.ts`
into another page and importing it back), check explicitly against the story's
**out-of-scope file list** before approving the refactor. Even a small,
mechanical constant-hoisting change is still a scope violation if it touches a
protected file. If an out-of-scope file's constant value is genuinely needed
elsewhere, define a **local, unexported duplicate** in the new location instead
of importing across scope boundaries. This avoids owning modifications to
files that are explicitly declared out-of-scope and prevents future rework when
the out-of-scope file changes. Example: STORY-30 needed the 12-Sunday horizon
value used by `availability/page.tsx`, but that file was out of scope — the fix
was to define `const SUNDAY_HORIZON = 12` locally in the home page, not import
it back from the availability module.

## Story Context accuracy verification (CHORE-19 lesson)

When authoring a story's Context section, descriptive claims about the current
codebase state (e.g., "The user's desktop screenshot of `/pt-PT/availability`
shows:...") must be verified against the actual, live code before refinement
approves the story. A mismatch between Context description and reality (e.g., a
screenshot describing what the *home page* looks like after a prior chore, not
the *availability page* this story addresses) is a distinct failure mode from
scope creep — it reflects factual inaccuracy in the story's own framing, not
overreach in the plan. Symptom: Refiner exploration uncovers that the Context's
described state doesn't exist in the current code. Mitigation: before approving
a story, Refiner should spot-check key Context claims by navigating the app
locally or reviewing the referenced component's actual JSX, not just trusting
the narrative. This catches stale/mismatched Context before Challenge and
Implementation waste cycles on a story based on false premises.

## Scope enforcement during Challenge and Review (CHORE-20 lesson)

When a story plan includes a design decision or judgment call that it explicitly
flags as "non-blocking" in its own "Open questions" or "Key design decisions"
section, that transparency does NOT exempt the decision from scope enforcement.
Challenge and Review personas must still independently verify the decision
against the story's declared acceptance criteria and out-of-scope list. The
plan author's transparency is good practice, but it does not transfer the
enforcement burden — that remains Challenge/Review's structural role. Symptom:
a plan that says "Design decision #3: I'm changing X, flagged as non-blocking
in Open questions #2" still requires Challenge to read AC3 and confirm whether
X actually stays in scope. Example from CHORE-20 (cycle 2 Challenge phase):
cycle 1's plan included a decision to move BUGFIX-*.md creation location from
top-level `docs/stories/` into `docs/stories/done/`, with a flagged note "this
is non-blocking." Challenge read AC3 (which explicitly asks only for numbering
logic to scan both directories, not for creation location to change) and
correctly reverted the decision as scope creep, even though the plan had been
transparent about making it. Second example: the same cycle 1 plan included a
decision to add BUGFIX-*.md to `status-glance.md`'s file-type scan, also
flagged. Challenge reverted this too — AC3 asked only for the path to cover
both directories, not for file types to expand. Keep the transparency in plans;
it surfaces reasoning and risks clearly. But do not treat plan flags as self-enforcement.

## Branch and commit discipline for out-of-repo and multi-file edits (CHORE-20 lesson)

**Out-of-repo edits require explicit rollback plan and PR disclosure:**
When a story's implementation plan includes edits to files outside the git
repo (e.g., `~/.claude/commands/*.md` global command definitions), implement
a rollback safety net because these files are not version-controlled by the
repo: (1) before editing, snapshot each file: `cp ~/.claude/commands/status-glance.md{,.bak}`;
(2) apply the edits; (3) verify the edits are correct by diffing against the snapshot:
`diff -u ~/.claude/commands/status-glance.md{,.bak}` (confirming only intended
changes, nothing extra); (4) delete the `.bak` files after verification (do not
leave them behind). Additionally, the PR description must explicitly disclose
these out-of-repo changes in plain language with file names and nature of the edits,
since the GitHub diff will not show them. This breaks the convention that "the PR
is the complete picture of all changes" — highlight it explicitly so reviewers
know to ask for a walkthrough. Example: CHORE-20 PR #49 note in description:
"Five global command files (`~/.claude/commands/{status-glance,stories,triage,fix,deliver}.md`)
were also updated to scan both `docs/stories/` and `docs/stories/done/` directories.
These are not visible in the diff but are required for the feature to work."

**Untracked files can silently bundle into commits:**
Before the first `git add` in a story's implementation phase, run `git status`
and check if any pre-existing untracked files are present (not `.gitignore`-d,
not staged, not part of the story's work). A careless `git add .` or `git add
docs/` can bundle these into the commit without the implementer noticing. At
Review time, the diff will show unexpected file changes that weren't part of
the story's plan, requiring rework. Mitigation: (1) Implementer runs `git
status` before the first broad `git add`, stashes or selectively ignores
pre-existing untracked files, then uses targeted paths like `git add
docs/stories/done/` instead of `git add .`; (2) Reviewer, upon seeing
unexpected files in the diff, asks the implementer to verify they were
intentional — do not approve diffs with mystery files. Example: CHORE-20
Implementation bundled an untracked file in its first commit; Review caught it
and Rework fixed it via `git reset --soft` + selective re-commit.

**Rewriting already-pushed commits on solo branches:**
When Review identifies unwanted changes in a commit that has already been
pushed to a solo feature branch (e.g., `chore/CHORE-20-...`), use `git reset
--soft HEAD~1` (not interactive rebase, which requires interactive terminal
input incompatible with automated environments) to undo the commit while
keeping its changes staged. Then selectively unstage the unwanted changes:
`git restore --staged <unwanted-file>`. Re-commit with only the intended files.
Force-push with `git push --force-with-lease` (never bare `--force`, which
silently loses remote tracking and can overwrite concurrent pushes). Example:
CHORE-20 Implementation notes document this pattern fixing the untracked-file
bundling issue in a second iteration.

## Stack notes

**Node.js and TypeScript:**
- When bumping the Node.js runtime version (in `.github/workflows/ci.yml` and `package.json`'s `engines` field), also bump `@types/node` to the same major version (e.g., Node 24 → `@types/node@^24`). TypeScript relies on `@types/node` for Node.js API definitions; mismatched versions cause false-positive type errors or incomplete intellisense.

**Next.js 16 + next-intl versioning:**
- Next.js 16 renamed `middleware.ts` → `proxy.ts`. The file **must** be named `proxy.ts`; `middleware.ts` is deprecated.
- `next-intl@^3` does not peer with Next.js 16; use `next-intl@^4`. The v4 API surface (e.g. `defineRouting`, `createNavigation`, `createMiddleware`, `getRequestConfig`) is identical for our usage.
- **`Link` from `@/i18n/navigation` works in both Server and Client Components** (STORY-16): next-intl v4's `createNavigation` factory exports conditional `react-server`/`react-client` conditional exports. This means the same `Link` import resolves correctly whether the importing component has `'use client'` or not — no wrapper component needed. Both `Server` and `'use client'` components can `import { Link } from '@/i18n/navigation'` directly and use it.

**Tailwind v4:**
- `create-next-app` with Next.js 16 installs Tailwind v4, which uses `@import "tailwindcss"` instead of the old `@tailwind base/components/utilities` directives. Do not replace with the old syntax.
- **Google Fonts subset support verification (CHORE-23)**: When adding a new `next/font/google` font with a non-standard subset (e.g., `latin-ext` for non-ASCII locales like Portuguese), verify the font family actually publishes that subset on Google Fonts **before** assuming build success. The build-time `next/font/google` fetch will fail silently if the subset name is misspelled or unsupported by that font. Mitigation: run `npm run build` in the implementation environment before marking PR ready — a full build surfaces subset support gaps that static code review cannot detect. Example: CHORE-23 flagged this as a risk for JetBrains Mono `latin-ext`; `npm run build` confirmed the subset exists and the feature works.
- **Dark mode class-driven strategy** (CHORE-11): In `globals.css`, add `@custom-variant dark (&:is(.dark *));` to force all `dark:` utilities to be class-driven (`.dark` class on root element) rather than system media-query-driven. Verify via compiled CSS that no `@media (prefers-color-scheme: dark)` blocks remain and all `dark:` selectors are gated by `:is(.dark *)`. This resolves component-level inconsistencies (e.g., button `dark:` responding to OS media query while theme state is class-driven) project-wide.
- **Semantic color tokens must be contrast-verified in both themes** (STORY-19): When adding new semantic token pairs to `:root` and `.dark` blocks (e.g., `--warning`/`--warning-foreground`), do not guess HSL values — measure actual contrast in compiled CSS via WebAIM or equivalent WCAG checker in both light and dark themes before finalizing. Example: an amber warning banner initially tuned to `hsl(32 95% 44%)` measured only 2.83:1 contrast ratio (too low for WCAG AA), required re-tuning to `hsl(32 95% 30%)` to reach 5.23:1. The token-based approach (values per theme scheme) is superior to raw Tailwind palette + hand-added `dark:` overrides because it centralizes the per-scheme values in one place, making regressions visible at review time. Verify contrast of both the token's text and background combinations in both themes before committing.
  - **Automated WCAG contrast regression testing for color tokens** (CHORE-17 pattern): Rather than one-off manual measurement, write a CI-safe e2e test that reads `app/globals.css` at test time, extracts HSL custom properties from both `:root` and `.dark` blocks via regex, converts HSL→sRGB→relative-luminance→contrast ratio using standard WCAG formulas, and asserts ratio ≥ 4.5 for WCAG AA compliance. This test automatically re-validates whenever a token's HSL value changes, preventing silent regressions. When implementing: (1) read globals.css via Node.js `fs` (no browser); (2) use regex to extract HSL triples (e.g. `--card: 0 0% 100%`); (3) implement standard HSL conversion with relative-luminance calculation; (4) compute contrast ratio as `(lighter + 0.05) / (darker + 0.05)`; (5) assert ≥ 4.5. This pattern is reusable for any future color-token pair that requires accessibility verification. Example: `e2e/card-ui-primitive.spec.ts` AC2 tests.
  - **Surfacing marginal-but-pre-existing contrast via test coverage**: When new component test coverage discovers a thin-but-passing contrast ratio on a token pairing that already shipped in production (e.g., `text-muted-foreground` on `bg-card` ≈ 4.83:1 in light theme, above the 4.5 AA floor but with minimal margin), document the exact ratio in the story's AC verification section and gate it with an automated test. This is not a bug to fix (it's pre-existing and passing); instead, it surfaces an architectural constraint that wasn't previously tested, and the automated test ensures future changes don't degrade it further without test failure. This approach balances rigor (measuring actual ratios, not guessing) with pragmatism (not changing working, shipped code). Example: CHORE-17 AC2 verification notes (story file) and `e2e/card-ui-primitive.spec.ts` test assertions.
  - **Solid-fill badge remediation for translucent-tint WCAG failures (CHORE-19 pattern)**: When a translucent-tint text color (e.g., `text-destructive` on `bg-destructive/10`) fails WCAG AA contrast, consider switching to a solid-fill badge (`bg-destructive text-destructive-foreground`) reusing an existing paired-foreground token, rather than retuning the shared base token (e.g., `--destructive`). The solid-fill approach avoids blast-radius to other components using the same token pair (e.g., error banners elsewhere on the app with the same combo). Contrast math: translucent `text-destructive` on `bg-destructive/10` can fail (example: 4.14:1 light / 1.93:1 dark), while solid `text-destructive-foreground` on `bg-destructive` comfortably passes (example: 4.62:1 light / 9.59:1 dark). When the target token pair already has a paired -foreground counterpart in use elsewhere (e.g., Button's `destructive` variant), reuse it directly — zero new CSS properties introduced. If no -foreground pair exists for a new semantic token, prefer creating one (via STORY-19's contrast-verified pattern) over introducing a one-off solid-fill color. This pattern applies to any color-status pair (destructive/destructive-foreground, warning/warning-foreground, etc.).
  - **Within-component consistency check on token-pair remediation (code-review heuristic)**: When fixing a WCAG contrast bug on a specific token pair within a component, scan the same component file for other uses of the same failing combination and apply the fix uniformly. This maintains visual consistency within the redesigned component and prevents "half-fixed" components where some text using the same failing tokens still fails contrast. Example: CHORE-19's blocked-row badge and error banner both used the failing `text-destructive` on `bg-destructive/10` combo — fixing only the badge would have left the banner visually inconsistent with the redesigned Card. Apply this check when reviewing changes that touch token pairs: `grep -r "text-<color> \|bg-<color>/10"` within the component's file to find all related uses.
  - **Alpha-blended color contrast rigor (CHORE-27)**: When using alpha-blended or translucent composites of color tokens (e.g., `hsl(var(--brand)/55%)` for a shadow over `--header`, or `bg-destructive/10` for a light tint), do not cite the pure token's contrast ratio (e.g., `--brand` vs `--header` at 7.4:1) as proof of the composite's visual clarity. The pure ratio does not represent the actual rendered composite color after alpha blending. **Correct approach**: either compute the actual composited HSL before citing a ratio, or skip the ratio claim entirely for decorative composites and rely on real-browser visual verification instead. This prevents misleading accessibility claims and ensures contrast measurements are representative of what users actually see. Applies to any future use of alpha-based colors (translucent overlays, semi-transparent shadows, tinted backgrounds).
  - **OKLCH→HSL conversion with documented pairing scope (CHORE-23 pattern)**: When a design mockup encodes a color in OKLCH (e.g., `oklch(0.74 0.17 55)`), convert via standard OKLab→sRGB→HSL matrices to a CSS custom property. **Measure actual contrast** for the intended use-case pairing (e.g., `--brand-foreground` on `--brand`) via the HSL→luminance→ratio helpers (CHORE-17 pattern, line 183–184). **Document the verified pairing** in the token's CSS comment: "verified: `token-a` on `token-b` = X.XX:1 (both themes); NOT verified as standalone text on page background." This prevents future stories from assuming broader contrast validity than tested — e.g., reusing a token verified only for bold accent pairs in small body text. Example: CHORE-23's `--brand` verified only as solid fill (8.31:1 with foreground), not as `text-brand` on `bg-background` (~2.4:1, failing) — any story wanting that use case must re-measure independently.
  - **Focus-visible ring contrast on new background surfaces (CHORE-28)**: When introducing a new non-`--background` surface token (e.g., `--header`, future `--sidebar`), audit every interactive element that renders on that surface for `focus-visible:ring-*` contrast. The shared `--ring` token is tuned for contrast against `--background` and may not meet WCAG 2.4.11/1.4.11 3:1 floor when the surrounding surface changes — example: CHORE-28 measured `--ring` at ~1.03:1 against `--header` in light theme (far below 3:1), while dark theme's `--ring` cleared at ~12.2:1. **Reusable fix pattern**: reuse the surface's own `-foreground` token as the ring color (via `focus-visible:ring-<surface>-foreground`), if it's already contrast-verified. This adds zero new CSS tokens and leverages existing measurements. Example: `focus-visible:ring-header-foreground` reuses the already-verified 16.38:1 light / 16.12:1 dark pair, comfortably clearing the 3:1 non-text floor in both themes. Apply uniformly to all interactive elements on the new surface, not just inactive states (the ring paints outside the button's border-box against the surrounding surface regardless of the button's own fill).
- **Client-side library state persisting across Next.js SSR remounts** (CHORE-13): Libraries like `next-themes` that manage DOM state imperatively via `useEffect` (e.g., applying `.dark` class) will have that state wiped when a parent Server Component remounts (e.g., when a dynamic segment like `[locale]` changes, triggering a re-render of `app/[locale]/layout.tsx`). The effect re-applies the state *after* paint, causing a visual flash. To fix: mirror the library state into a first-party `document.cookie` (synced by a client component with `useEffect` watching the library's resolved state), then read that cookie server-side and render the element with the correct class directly in the SSR markup. The cookie uses `Secure; SameSite=Lax; path=/; max-age=<1year>` — note that `Secure` works on `http://localhost` due to the browser's potentially-trustworthy-origin exception, so this pattern is safe in local dev. The library's blocking inline script (next-themes) remains authoritative for cold loads (before any client JS runs); the cookie becomes authoritative for soft navigations where the script doesn't re-run. Pattern: `lib/theme/theme-cookie.ts` (constants), `components/ThemeCookieSync.tsx` (`'use client'` component syncing library state to cookie), `app/[locale]/layout.tsx` (read cookie via `await cookies()` and render element with class). See CHORE-13 for the worked example.
- **Checked/selected state overriding hover utilities** (BUGFIX-03): When adding a "selected" or "checked" visual state via `has-[:checked]:` that must override an existing `hover:` utility on the same element, use the compound selector `has-[:checked]:hover:bg-X` (specificity 0,3,0) instead of only the base `has-[:checked]:bg-X` (specificity 0,2,0). The base selector alone ties in specificity with the existing `hover:bg-accent` utility, creating a source-order-dependent race that varies across browsers. The compound selector has deterministically higher specificity and makes the override robust. Example: a skill-level radio button that displays with a highlight when selected must use both `has-[:checked]:bg-primary has-[:checked]:text-primary-foreground has-[:checked]:hover:bg-primary` on the `<label>` wrapper. Confirmed via direct Tailwind v4 pipeline compilation and headless-Chromium rendering that CSS cascade math, not visual reasoning alone, determines the output. Apply this pattern retroactively to any existing `has-[:checked]:` utilities that share an element with `hover:` utilities.
- **Compiled CSS output location under Turbopack**: `.next/static/chunks/*.css`, not `.next/static/css/*.css` (webpack-era assumption). Relevant for filesystem-based CSS assertion tests.
- **Native form control theming via `color-scheme` (BUGFIX-04)**: The `color-scheme` CSS property governs native browser-rendered UI for `<select>` popups, `<input type="date|range|color|radio|checkbox">` ring/dot painting, scrollbars, and other browser chrome. **This is inherited** from the root element, so set it once globally in `app/globals.css` via `:root { color-scheme: light; }` and `.dark { color-scheme: dark; }` — NOT the shorthand `color-scheme: light dark;` on `:root` alone. The shorthand tells the browser "pick based on OS preference," which re-couples native widgets to `prefers-color-scheme` media queries, undoing the CHORE-11 class-driven dark-mode strategy. Explicit per-class values keep native theming tied to the same `.dark` class signal. When auditing a bug report on one native control (e.g., `<select>` low contrast), search for ALL native control types and `type=` variants app-wide via grep (radio, checkbox, date, range, color, search, number, file, plus bare `<input>` and `<select>` elements), because `color-scheme` affects their appearance cascade-wide once set. This is a case where reasoning about CSS changes requires auditing inherited/global properties' full blast radius, not just the one component mentioned in the bug.
- **Native `<select>` popup dark-mode limitation on Windows Chromium (BUGFIX-05 correction)**: `color-scheme` reliably affects the *closed* select control's colors but **does not reliably control the open popup's `<option>` background rendering on Windows**, even though the inherited text color does follow the scheme (https://issues.chromium.org/issues/41006289, https://issues.chromium.org/issues/415953695 — genuine, unresolved Chromium platform gaps, not an app-level bug). If dark-mode contrast in the **open popup** is critical and `color-scheme` CSS-only verification shows no issues on Linux but real Windows end-users report unreadable text, **the durable fix is to replace the native `<select>` with a custom-styled DOM-rendered dropdown** (e.g., shadcn/ui `Select` using Radix UI primitives). This eliminates the entire class of OS-delegated-popup-rendering bugs because the popup is now a normal DOM node with CSS that Blink renders identically across platforms. Keep the `color-scheme` declarations in place (they are correct for other native controls like date inputs); only replace the specific native control that the user reported as broken. When adding new native controls in future stories (e.g. `<input type="date">`), apply the same decision gate: if dark-mode contrast matters for that control's popup and `color-scheme` CSS-only testing is insufficient, prefer a custom-styled Radix primitive from the start rather than chasing CSS mitigations for native rendering later.
- **Shrink-to-fit trailing column for right-aligned table columns** (STORY-14): When an HTML `<table>` uses the browser's default `auto` layout (no `table-layout: fixed`) and needs a trailing column (e.g. actions) right-aligned at the table's edge without magic pixel widths, apply `w-[1%] whitespace-nowrap text-right` to both the `<th>` and `<td>` of that column. The `width: 1%` tells auto-layout to give the column only the bare minimum width; `white-space: nowrap` prevents the content from wrapping or shrinking below its intrinsic width, forcing auto-layout to assign remaining space to unconstrained (wider) columns. The result: the trailing column always stays shrink-to-fit and right-aligned at the table's trailing edge, regardless of translation string length or viewport width. This is more robust for i18n'd tables than hardcoded pixel widths with `table-layout: fixed`. Example: `<th className="w-[1%] whitespace-nowrap px-4 py-3 text-right font-medium">Actions</th>`.
- **Tailwind v4 `cva` size/variant redeclaration landmine with `twMerge` last-occurrence-wins** (CHORE-24): When using `class-variance-authority` (cva) to compose a Tailwind utility-based primitive (e.g., Button, Badge), and a `size` or `variant` option redeclares a class group that the base already covers (e.g., base has `rounded-full`, and `sm` size variant includes `rounded-md`), `twMerge` (used by `cn()` from `lib/utils.ts`) keeps only the **last occurrence** of that class group in the final composed string. This means `sm` or `lg` buttons will silently keep the size variant's `rounded-md` instead of inheriting the base's `rounded-full` — appearing square-cornered while the default size is pill-shaped, a subtle AC violation ("all variants/sizes updated"). **Mitigation**: audit every cva-based primitive's base and size/variant strings before committing. For any shared Tailwind class group (rounded, bg, text, etc.) that appears in both base and a size/variant, remove the redeclaration from the size/variant — the base's value will be inherited, and `twMerge` will compose without collision. Pattern: verify base has `rounded-full`, then confirm neither `sm` nor `lg` variant strings repeat any `rounded-*` class. This applies to any future cva-based primitive edit (Button, Badge, future Separator, etc.) — it's not a one-off issue but a reusable audit discipline for `twMerge`'s deterministic last-occurrence behavior. Example: `components/ui/button.tsx` (CHORE-24) removed redundant `rounded-md` from `sm` and `lg` size variant strings to prevent the landmine.
- **Tailwind v4 `calc()` in arbitrary values requires space-encoding (CHORE-22)**: When using CSS `calc()` in a Tailwind arbitrary value, literal spaces around `+`/`-` operators must be encoded as underscores per Tailwind's arbitrary-value syntax (e.g., `pb-[calc(110px_+_env(safe-area-inset-bottom))]`). A class like `pb-[calc(110px+env(...))]` without the underscores is invalid CSS (`calc()` requires whitespace around operators) and Tailwind will either drop the utility or emit broken CSS. Verify the compiled CSS under `.next/static/chunks/*.css` (Turbopack output location, not webpack's `.next/static/css/`) to confirm the calc is correct, since a green `npm run build` exit code is not sufficient — build success only means no syntax errors in the config, not that the generated CSS is valid. This pattern applies to any future Tailwind class using `calc()` with environment variables, custom properties, or multiple operands.

**Position:fixed DOM placement affects keyboard tab order (CHORE-22):**
- A `position: fixed` element's visual position is independent of its DOM position — CSS removes it from the layout flow. However, **DOM position still determines keyboard tab order**, which does not follow CSS layout, but rather the HTML tree order. If a `position: fixed` navigation bar is placed last in the DOM tree (after all page content), keyboard users must Tab through every form, link, and button on the entire page before reaching the navigation, directly violating WCAG 2.4.3 (focus order). **Fix**: place the fixed element early in the DOM tree, right after the header or first persistent chrome. Example: `<AppHeader />`, `<BottomNav />` (fixed, must be early), `{children}` (page content). This ensures the navigation is reachable in a bounded number of Tab presses regardless of page length. A side-effect: visually, the fixed bar renders on top of content due to `z-index`, but logically, it's announced early in the document tree. Verify keyboard reachability by manually Tab-walking from the top of a content-heavy page (e.g., a table with many rows); the fixed element should be reachable before the first row of content, proving DOM position is correct.

**HTML tables with inline-edit actions (STORY-14):**
- When a table row must support inline-edit mode where an `<input>` stays in one column (e.g. name) but its Save/Cancel buttons must move to a different column (e.g. actions, to avoid layout jump), use the HTML5 `form` **attribute association** instead of trying to nest a `<form>` tag across multiple `<td>` elements (which the HTML parser will foster-parent out of the table, silently breaking layout). Pattern: place the `<input>` in the first `<td>` with `form={editFormId}` attribute, then render a `<form id={editFormId}>` wrapping the Save/Cancel buttons in the target `<td>`. Each element remains a direct child of its own `<td>`, no foster-parenting risk, and Enter-to-submit behavior is preserved because the input's `form` attribute lets the browser associate it with the form in the other cell. Example: `<input form="pm-edit-form-123" ... />` (name column) paired with `<form id="pm-edit-form-123"><button type="submit">Save</button></form>` (actions column).

**shadcn/ui + Tailwind v4 integration:**
- `components.json` must have `"tailwind": { "config": "" }` (empty string, not a file path). The CLI auto-detects Tailwind v4 from `postcss.config.mjs`. If detection fails, create `components.json` manually.
- `npx shadcn@latest init` and `npx shadcn@latest add` may be blocked in sandboxed environments. In that case, create `components.json`, `lib/utils.ts`, and component files manually — the CLI output is deterministic and identical.
- Tailwind v4 CSS token layer: shadcn injects two `@layer base` blocks (custom properties in `:root` and property defaults on `*`/`body`) plus an `@theme inline` block mapping CSS variables to Tailwind utilities. Multiple `@layer base` declarations in one file are valid and merge correctly. The `@import "tailwindcss"` line must remain first.
- **shadcn UI primitive house pattern (Button, Select, Card, future Badge/Separator)**: All shadcn primitives in this repo follow a consistent pattern, regardless of complexity. (1) Plain functions, not `React.forwardRef` — function components are sufficient for both Radix wrappers and simple div wrappers. (2) Root element has `data-slot="<name>"` attribute for testing/styling hooks. (3) Styling via `cn()` from `lib/utils.ts` composing Tailwind utilities; no inline styles, no CSS-in-JS. (4) Props spread via TypeScript's `React.ComponentProps<"div">` (or the wrapped Radix primitive's ComponentProps) for full HTML/Radix attribute compatibility. (5) Multiple named exports via `export { Component1, Component2, ... }` matching the upstream shadcn names, not default exports. (6) `'use client'` **only** if the component uses Radix UI primitives (which are client-only) or React hooks (useState, useEffect, etc.). Pure div-wrapper primitives (Button without variants, Card, CardHeader, etc.) do **not** need `'use client'` and can be imported directly from Server Components. When adding future primitives (Badge, Separator, Divider, etc.), follow this pattern to maintain consistency.
- `components/ui/button.tsx` requires `'use client'` as its first line because it uses Radix `Slot` (a client-only primitive) for the `asChild` feature, even though Button itself is technically a plain wrapper. Importing into Server Components is valid — Next.js treats it as a client boundary automatically.
- **Native `<details>`/`<summary>` as a lightweight dropdown**: When shadcn `DropdownMenu` is not installed and no React state is required (e.g., a simple show/hide panel), native HTML `<details>`/`<summary>` works in an async Server Component without `'use client'`. The browser handles open/close natively; `<summary>` receives `cursor-pointer` via CSS. As of STORY-13, click-outside and Escape dismissal are handled via a thin `'use client'` wrapper (ref + document-level `click`/`keydown` listeners) around the `<details>` node — see `components/UserWidgetMenu.tsx` for the reference pattern. The native element itself still requires no `'use client'` boundary for static/non-dismissible uses; add the wrapper only when outside-click or Escape dismissal is required. The outside-click listener must be attached on the plain bubble-phase `'click'` event — not `'mousedown'`, not `{ capture: true }` — so that `<summary>`'s native open/close toggle (which runs as the event's default action, after bubble-phase listeners) is not fought by the outside-click handler. When dismissing via Escape, restore focus to the trigger element (ARIA APG disclosure widget convention) to preserve keyboard user context.
- **Role-gated layout components return `null` when empty** (STORY-16): When a component conditionally renders a landmark region (e.g., `<nav>`, `<main>`) based on user role, and the role has zero items to display, return `null` (omit the element entirely) instead of rendering an empty landmark. Empty landmark regions are an ARIA anti-pattern — screen-reader users navigating by landmark list find nothing inside them. Example: `AppNav` returns `null` when `role !== 'admin'`, not an empty `<nav aria-label="..."><ul></ul></nav>`.
- **Wrapping existing heading elements inside shadcn primitive title slots (CHORE-18)**: When a page or component has an existing `<h1>`/`<h2>`/etc. element and it must be moved inside a shadcn primitive's title slot (e.g., `CardTitle`), remember that `CardTitle` and similar subcomponents render as plain `<div>` with no semantic heading role (per the shadcn UI primitive house pattern above). Naively replacing `<SomeHeading>{title}</SomeHeading>` with `<CardTitle>{title}</CardTitle>` silently drops the heading element entirely — only the text content is preserved, breaking screen-reader heading-navigation (cf. STORY-16's "empty landmark is an ARIA anti-pattern" precedent). **Fix**: nest the original heading element *inside* the primitive's title slot, e.g. `<CardTitle><h1 className="text-2xl font-semibold">{title}</h1></CardTitle>`, keeping the heading's own sizing classes on the inner element (not on `CardTitle`, whose default classes alone would under-size it). **Regression testing**: add scoped `getByRole('heading', { level: N })` assertions to both the smoke and integration test suites, anchored to the testid'd container so assertions fail loudly if the heading is accidentally dropped in a future edit (only `getByRole('heading')` catches this — lint, TypeScript, and build passes will not). Example: `e2e-integration/home.spec.ts` assertions added in CHORE-18 cycle 2.

**Locale layout pattern:**
- `app/layout.tsx` must be a minimal passthrough (no `<html>`/`<body>`); the locale layout `app/[locale]/layout.tsx` owns `<html lang={locale}>`, `<body>`, `NextIntlClientProvider`, and all persistent chrome (header, nav).
- **Global CSS must be imported in `app/layout.tsx`** — add `import './globals.css'` there. The locale layout does not import it. Without this, Tailwind generates no CSS and the app is completely unstyled in production.
- Pages only render their own content; shell belongs in the locale layout.
- Call `getMessages({ locale })` with the validated locale string, not `getMessages()` without arguments.
- **Route groups for conditional chrome**: In Next.js App Router, nested layouts **compose inside** their parent — they do not replace or suppress parent content. To hide app chrome (e.g., `AppHeader`) on specific routes (login, marketing pages), use a route group `(app)/`. Move chrome-rendering code into `app/[locale]/(app)/layout.tsx` and move app pages into `app/[locale]/(app)/`. Routes outside `(app)/` (e.g., `app/[locale]/login/`) will not render the chrome. Route groups are invisible to URLs — `/[locale]/`, `/[locale]/admin/users`, etc. remain unchanged. This pattern prevents trying to use nested layouts to suppress parent content, which is not possible in Next.js. See STORY-10 and ADR-009 for context.
- **Same-route-group client navigation does not remount ancestor layouts** (STORY-21): When two routes are in the same route group (e.g., `/pt-PT/` and `/pt-PT/settings` both inside `(app)/`), client-side navigation via `next/link` does not unmount or remount the shared ancestor layout (e.g., `AppHeader`, `UserWidgetMenu`). The layout instance persists. If a form component in one route mutates data that the persistent layout reads (e.g., updating `display_name` that `AppHeader` displays), the layout will show stale data until `router.refresh()` is called after the mutation succeeds. `router.refresh()` (from `@/i18n/navigation`) re-invokes Server Component render functions without a full page reload, allowing the persistent layout to re-fetch and reflect the new state. This contrasts with cross-group navigation (e.g., `/login` outside `(app)/`), which unmounts the whole layout tree. Pattern: after a successful mutation in a same-group Client Component form, call `router.refresh()` to refresh all ancestor Server Components in the layout, then update local form state. Example: `components/DisplayNameForm.tsx` (STORY-21) calls `router.refresh()` after a successful PATCH, allowing the persistent `AppHeader` to show the new name without a full reload.

**i18n lazy-loading pattern in async Server Components:**
- In async Server Components with multiple role-based early returns (e.g. `home/page.tsx`), load translation namespaces **lazy** — only call `await getTranslations(...)` just before the branch that uses them. This avoids paying for namespaces that are never used on short-circuit paths. Example: if a member request returns early, the Admin-only `Home` namespace is never fetched.

- **Tap targets (WCAG 44px minimum)**: Use `min-h-[44px]` (not `h-11`) for interactive elements. `h-11` (2.75rem) computes to exactly 44px only at the browser's default 16px/rem. `min-h-[44px]` is a hard pixel floor that holds regardless of font scale and makes tests flake-proof. When an interactive element contains a decorative icon or avatar, apply `min-h-[44px]` to the outer interactive element (e.g., `<summary>` or button), not to the inner visual component — the inner component can be smaller (e.g. `h-8 w-8` for an avatar) for compactness while the outer element provides the full tap target.
- **Multi-reporter pattern**: Set `reporter: [['html'], ['list']]` in `playwright.config.ts`. The `html` reporter generates the detailed report with screenshots; `list` preserves console output in CI logs for quick debugging. Use this pattern, not `[['html']]` alone.
- **Test-location-based CI job routing via `testDir` (CHORE-15)**: Playwright test discovery is configured via `testDir` in each config file: `playwright.config.ts` (testDir: './e2e') discovers the smoke suite for the `lint-build-test` job, and `playwright.integration.config.ts` (testDir: './e2e-integration') discovers the integration suite for the `integration-test` job. When migrating a test file between suites (e.g., from placeholder-credential smoke tests to local-Supabase-backed integration tests), moving the file's physical location automatically re-routes it between CI jobs **without requiring any changes to `.github/workflows/ci.yml`**. No explicit CI job routing needed — file location alone determines which Playwright config will discover it. Example: CHORE-15 migrated `e2e/app-nav.spec.ts` → `e2e-integration/app-nav.spec.ts`, which automatically moved the tests from the smoke job to the integration job. This pattern is reusable for future test migrations; no special CI wiring is required.
- **OS-delegated rendering vs. DOM/CSS verification confidence (BUGFIX-05)**: When assessing whether a feature's Linux-sandbox Playwright verification will generalize to real Windows/Mac end-user machines, the key decision point is **whether the feature uses OS-delegated native rendering** (e.g., native `<select>` popups, native date/time/color pickers, OS scrollbars, system context menus). If yes — the feature relies on browser-delegated OS-level rendering that may diverge across platforms — Linux sandbox verification is **LOW confidence** and a real cross-platform spot-check is load-bearing, not optional. If no — the feature is ordinary DOM/CSS (even if Portal-rendered) that Blink renders identically across platforms — Linux verification is **HIGH confidence** and will generalize. Example: BUGFIX-04's `color-scheme` CSS fix *appeared* correct on Linux Chromium but failed on real Windows Chrome; BUGFIX-05's replacement with a custom Radix `Select` component moved from OS-delegated (low confidence) to DOM/CSS (high confidence), materially improving verification generalization. Apply this heuristic when deciding whether a feature flagged "not testable in this sandbox" needs a real spot-check or can rely on CI automation.
- **Artifact uploads — `if-no-files-found` gate selection (CHORE-05)**: For deterministic output directories (e.g. `playwright-report/` from html reporter), set `if-no-files-found: error` in `actions/upload-artifact@v4`. This creates a real CI gate and prevents silent failures. **However, for test-results directories from API-only/headless Playwright suites** (no page rendering, no visual artifacts captured even with `screenshot: 'on'` — which is itself a no-op), use `if-no-files-found: warn` instead — a fully-passing API-only run produces only hidden `.last-run.json` metadata, and `upload-artifact@v4` excludes hidden files by default, so `error` fails on every green run. Reserve `warn` for optional/conditional outputs that are legitimately empty on success.
- **`if: always()` placement in CI**: In GitHub Actions, place `if: always()` at the step level (same indent as `uses:`), not at the job level. Correct placement ensures artifacts upload even when earlier steps fail.
- **CI-config changes require real GitHub Actions verification (CHORE-05)**: When modifying CI workflow YAML or Playwright reporter config, a local-only test pass (e.g., `npm run test:e2e` locally, or Docker-based CI emulation) is insufficient to prove correctness. Actions-specific behaviors (e.g., `actions/upload-artifact@v4` excluding hidden files by default, or `$GITHUB_ENV` not unquoting values) only manifest on the real GitHub-hosted runner. Always poll the real CI run (`gh run view`, `gh pr checks`) after pushing a CI-config change to verify the job passes end-to-end before marking ready for review. This caught CHORE-05's artifact-upload gate misconfiguration before it reached full CI verification.
- **Fault-injection QA verification for database-layer test-infra (CHORE-14)**: When a test-infra story adds regression coverage for a database/permissions bug (e.g., missing GRANT causing 500 errors), validate the test suite actually catches the regression by fault-injection: (1) intentionally break the thing the test guards (e.g., `REVOKE SELECT, INSERT, UPDATE, DELETE ON public.roles FROM service_role;`), (2) re-run the test suite, (3) confirm tests fail with the expected error (e.g., 500 instead of 201), (4) restore the fix (re-apply the GRANT), (5) confirm all tests pass again. This proves the test suite is non-vacuous and genuinely detects the regression class, not just passing in the happy path. This pattern is especially valuable for database-layer coverage where regressions manifest as HTTP status mismatches, not assertion failures. Document this verification step in the story's AC or manual notes so reviewers understand the testing is rigorous, not accidental.
- **`__next-route-announcer__` strict-mode collision**: Next.js App Router always injects `<div role="alert" aria-live="assertive" id="__next-route-announcer__">` into every page for screen-reader route announcements. Any test using `page.locator('[role="alert"]')` will match this element AND any application error divs, triggering Playwright's strict-mode violation when there are 2+ matches. **For dynamically injected error messages, use `aria-live="polite"` instead of `role="alert"`**. Combined with `data-testid="<error-name>"` for test targeting: `<div data-testid="um-error" aria-live="polite">`. This avoids the collision entirely while maintaining screen-reader announcements. Add `data-testid` to all interactive auth components (buttons, forms) from the start to avoid brittle text-based selectors.
- **Form input accessible labels (WCAG SC 1.3.1)**: Every `<input>` in a form must have a programmatically associated label. Using a visual `<h2>` or `<label>` text is not sufficient. Add `aria-label={t('keyName')}` directly to the input, or use `<label htmlFor="inputId">` + matching `id` on the input. Placeholder text disappears on focus and does not satisfy the criterion. **Multi-input rows** (e.g., a table row with separate name + slots inputs for both add and edit): each input must have a **distinct** `aria-label`, not a shared label reused across inputs — a single label on multiple inputs violates SC 1.3.1. Pattern for add-row: `<input aria-label={t('namePlaceholder')} .../>` + `<input aria-label={t('slotsPlaceholder')} .../>`. Same distinct labels apply in edit mode — never fall back to placeholder-only labelling (STORY-17). **Grouped radio inputs** (STORY-11): Wrap a list of radio inputs in `<fieldset><legend>{t('question')}</legend>...</fieldset>` so assistive tech announces them as one question, not a flat list of unrelated labeled inputs. Each radio within the group is still individually labeled via `<label htmlFor>` or `aria-label`, but the group itself has semantic structure via the fieldset/legend pair. This matches native HTML semantics with zero extra ARIA and ensures screen-reader users understand the radio buttons are mutually exclusive choices within a single question. Do not render radio inputs as bare siblings with only individual labels and no grouping container. Example: `components/ClaimPersonForm.tsx` (STORY-11) lines 96-118.
- **`aria-label` composition for identity widgets**: When a button or interactive element shows multiple pieces of information (e.g., user's name + a menu affordance), compose the `aria-label` to include all of it: `{displayName} — {t('menuLabel')}` (e.g., "Simão — Menu de utilizador"). A bare `t('menuLabel')` overrides the computed accessible name from visible text and hides the user's identity from screen readers. Always preserve the semantic data in the accessible name.
- **Auth-gated test pattern**: When e2e tests require Supabase auth that CI cannot provide, gate skipping on an env var: `test.skip(!process.env.E2E_WITH_AUTH, 'reason')`. This skips the test in CI (no env var) but allows developers with `E2E_WITH_AUTH=1` and real credentials to run it locally. Never use `test.skip(true, ...)` with a boolean literal — it permanently disables the test everywhere, even locally. Always bind to an env var for local opt-in.
- **Playwright auth-cookie fixture pattern for local Supabase integration tests (CHORE-05)**: When writing tests against authenticated API endpoints that require session cookies, construct a Playwright fixture by reusing `@supabase/ssr`'s `createServerClient` + `setAll` callback to capture real session cookies from an actual `signInWithPassword()` call, rather than hand-rolling the cookie name/encoding scheme. This guarantees the cookies match what `lib/auth/guard.ts` expects without depending on internal `@supabase/ssr` cookie chunking/encoding details. Pattern: build a throwaway `createServerClient` with a `cookies.setAll` that just collects `{ name, value }` pairs, call `supabase.auth.signInWithPassword({ email, password })`, then use the collected pairs as the `Cookie` header for the API request (via `APIRequestContext` or `browser.newContext()`). This is test-only infra and does not apply to production auth flows. See `e2e-integration/fixtures.ts` (CHORE-05) for a worked example.
- **Playwright APIRequestContext string-payload Content-Type gotcha (STORY-25)**: When using `APIRequestContext.post(url, { data: 'raw string' })` to test malformed-JSON-body error paths, the raw string is only JSON.stringify-wrapped (turning `"not json"` into the JSON string literal `"not json"`, which parses fine) **if a JSON Content-Type header is set**. With no explicit Content-Type override, the raw, non-JSON-parsable string is sent as-is to the server, exercising the invalid_json branch. Pattern: `const response = await request.post('/api/path', { data: 'not json' })` — no Content-Type header — sends a literal `not json` body that `request.json()` will throw on, triggering the 400 `invalid_json` handler. Example: `e2e-integration/blocked-dates.spec.ts` lines 238-258 uses this to test AC5.
- **Do NOT create test fixture pages under auth-exemption paths (BUGFIX-02 critical finding)**: Never create QA scaffolding or e2e fixture pages at any URL path that matches the proxy.ts auth-exemption regex (e.g., `/login/`, `/auth/`). Any page nested under such a path will inherit the exemption and become silently reachable with zero authentication in production (e.g., a fixture at `app/[locale]/login/e2e-fixture/*/page.tsx` matches the `isLoginPath` regex in proxy.ts and ships as an unauthenticated public route, not caught by CI or static analysis). **Attempted non-viable approaches for CI-safe component testing** (documented so future implementers don't waste a cycle): (1) `page.setContent()` + `renderToStaticMarkup()` from Playwright's `.tsx` files — Playwright's own test transform applies a JSX pragma to all `.tsx` module loads (not just test files) that is incompatible with `react-dom`'s runtime, causing immediate SyntaxError; (2) `NODE_ENV === 'production'` guarding — this project's CI runs `next build && next start`, meaning `NODE_ENV=production` in CI too, so this guard would disable the test in CI, defeating its purpose. **Fallback pattern that works**: CI-safe static source guard — read the component file directly via Node.js `fs.readFile`, use a single full-string regex to assert the expected className is present (not independent substring checks, to catch typos like `sm:flex-no-wrap`). Example: `e2e/people-table-alignment.spec.ts` lines 261-276. This provides regression coverage without auth, without a live render, and without adding unauthenticated fixture routes to production.
- **Overflow measurement (DOM scrollWidth)**: For viewport overflow checks in Playwright, always use `document.documentElement.scrollWidth` (the `<html>` element), not `document.body.scrollWidth`. Overflow can sit on the root element when `body` doesn't extend there, giving false negatives. This matches the `smoke.spec.ts` pattern and is the correct canonical measurement.
- **`toHaveURL(RegExp)` matches the full URL including origin, not the pathname** (STORY-16): Playwright tests `expected.test(url.href)` against the entire URL string including protocol and host (e.g., `http://localhost:3000/pt-PT/admin/people`), not just the path. A left-anchored regex like `/^\/pt-PT/` will never match because the string starts with `http://`, not `/`. Use **right-anchored patterns only** for meaningful assertions: `/\/pt-PT\/?$/` matches the full href if it ends with `/pt-PT/` or `/pt-PT`, ruling out sibling routes like `/pt-PT/admin/people`. Omitting the `^` anchor is the correct pattern here.
- **Testing `router.refresh()` requires client-side navigation, not `page.goto()`** (STORY-21): When testing that a Client Component mutation properly refreshes a persistent ancestor Server Component (e.g., after a form saves, the layout header shows new data), the test must navigate via `page.click()` on a `next/link` Link, not via `page.goto(url)` or `page.reload()`. `page.goto()` always performs a full browser navigation that re-renders all Server Components regardless of whether `router.refresh()` exists, giving zero regression coverage — the test would pass even if the refresh call were deleted. Instead, use client-side navigation: submit the form, observe the mutation result, then click a Link to another route in the same route group. The persistent layout is not remounted by this client-side nav, so it will only show the new value if `router.refresh()` was actually called and fetched the updated data. Example: in `e2e/settings-display-name.spec.ts` AC3, after saving a new name, click the "Escala" wordmark (a Link to `/`) — the assertion that the header shows the new name is only meaningful because `router.refresh()` in `DisplayNameForm.handleSubmit` was called before this click.
- **Test literal acceptance-criterion scenarios, not proxy code paths** (BUGFIX-03): When an acceptance criterion specifies "renders correctly on initial page load with a pre-existing saved value," the test must use API/DB seeding (e.g., `page.request.put(...)` or manual DB mutation) **before** any `page.goto()` call to set up the initial state server-side, not click-to-reach-state-then-assert in a single browser session. Server-rendered initial props (from first `page.goto()`) and client-side optimistic state (from a click interaction in the same session) exercise genuinely different code paths in React/Next.js. A regression in one would not be caught by testing only the other. Pattern: create/seed the desired data via API, then navigate to the page and assert the visual indication is present. This ensures the Server Component's render and Client Component's interactive-state paths are both tested, not just one of them. This gap is especially visible in radio/checkbox UI where the DOM `checked` attribute may be correct but the visual affordance is missing.
- **Invisible-mechanism bug test coverage split** (CHORE-13): For bugs in transparent mechanisms (e.g., theme flash during cross-locale soft navigation), split test coverage across two approaches: (1) CI-safe assertion of the SSR/server-side fix in isolation — e.g., raw HTTP fetch with a cookie header, asserting the returned HTML has the correct class in the markup — proves the mechanism itself without needing auth. (2) Auth-gated visual reproduction — e.g., `E2E_WITH_AUTH`-gated `requestAnimationFrame` sampling of DOM state during the real user action, asserting the visual bug is gone. This split avoids needing to expose the mechanism on a public route (which may violate scope) while still capturing both the fix's mechanics and its user-visible effect. Pattern: use `page.request.get(url, { headers: { Cookie: '...' } })` for CI-safe SSR assertions, and `E2E_WITH_AUTH`-gated tests for real visual reproduction via rAF sampling.
- **Auth-gated test fixture cleanup with worker isolation** (STORY-14): When writing auth-gated tests that create temporary rows/fixtures in the database (e.g. a test person in the people-management table), use `test.beforeEach` with `testInfo.workerIndex` to generate unique fixture names per worker, avoiding cross-worker name collisions if tests run in parallel: `let testPersonName: string; test.beforeEach(({}, testInfo) => { testPersonName = \`Test Name (w${testInfo.workerIndex})\`; });`. Then use `test.afterEach` to unconditionally remove the fixture after every test (pass or fail), preventing leaks into the live data: `test.afterEach(async ({ page }) => { await removeFixtureIfPresent(page, testPersonName); });`. This pattern is essential for local developer testing where tests may run in parallel without CI's sequential constraint.
- **E2E fixture hygiene when adding uniqueness constraints** (STORY-11): When a story adds a new DB-level uniqueness constraint (e.g., a partial unique index on `linked_user_id`), audit existing e2e tests for any that mutate the constrained column without fixture isolation. A previously-"harmless" test mutation (e.g., linking a shared E2E_WITH_AUTH account to a person without cleanup) becomes a cascading test failure once the uniqueness constraint is enforced — the second test run fails because the first test's link was never cleaned up, and the constraint forbids re-claiming the same person. Even if the test uses an admin UI path that's not directly testing the constraint, the fixture-isolation pattern from STORY-14 must apply: create worker-isolated fixtures, verify state via read-back, and unconditionally clean up in `finally` blocks or `test.afterEach`. When adding a uniqueness constraint, also add a regression test that exercises the constraint's error code (e.g., 409 `already_linked` when claiming a second person with one account) to prevent silent regressions. See `e2e/claim.spec.ts` lines 392-419 for the regression test pattern and lines 15-27 (header comment) for fixture hygiene on an auth-adjacent self-service flow.
- **Test helper collision guard on non-unique DB columns** (STORY-08): When an e2e/integration test helper resolves a database row by matching a non-unique column (e.g., `display_name` with no DB uniqueness constraint), filter to all candidates and throw loudly if more than one matches, **before** using the result for any mutating operation. Example pattern: `const matches = users.filter(u => u.display_name === targetName); expect(matches.length).toBe(1); const self = matches[0]`. Silent collisions on non-unique columns risk tests mutating the wrong row — e.g., a self-demotion test accidentally demoting a different real admin account in a shared dev Supabase project. Apply this guard even when the test author believes a collision is impossible in the current environment — environments change, and the guard costs negligible performance.
- **Flexbox wrap for horizontal overflow at 375px (STORY-23, STORY-18)**: Horizontal overflow of action buttons/links at 375px viewport happens when Flexbox's default `min-width: auto` on ancestor flex items prevents non-shrinking children (e.g., buttons with `whitespace-nowrap shrink-0` from `buttonVariants`) from wrapping. Fix: add `min-w-0` to the flex container and `flex-wrap` on the `<ul>` / action-row `<div>`, so children wrap onto new rows at narrow viewports instead of overflowing. Example sites: header nav buttons (`AppNav.tsx`, STORY-23), table action buttons in view mode (`PeopleTable.tsx` lines 236, STORY-18). When adding a new item to such a list, no extra width-budget work is needed — just verify at 375px that wrapping still looks visually clean. **Residual limit**: this wrap-based fix only rescues *multiple* items wrapping onto more rows; a single item wider than the viewport on its own (e.g., a very long translated string) would still overflow, since `whitespace-nowrap` prevents internal wrapping — such a case would need a separate fix (truncation, shortening, etc.), not just more wrapping room.
- **Breakpoint-gated flex-wrap (BUGFIX-02 refinement)**: When a multi-button action row uses `flex-wrap` to handle narrow viewports, gate the wrap with a breakpoint to prevent unconditional wrapping at wider viewports where space is available. Pattern: `flex flex-wrap sm:flex-nowrap` (Tailwind's `sm` = 640px) allows wrapping only below the breakpoint, never at desktop/tablet. Without the nowrap gate, a shrink-to-fit column's auto-layout sizing can interact poorly with flex-wrap, causing the flex container to wrap unconditionally even with ample horizontal space. This was discovered in BUGFIX-02 when `PeopleTable.tsx`'s three-button actions row stacked vertically at 1280px despite ~1000px available. Confirmed via direct Tailwind pipeline + headless browser rendering before the fix — not static reasoning alone. The `sm:flex-nowrap` gate has no impact on the 375px case (`flex-wrap` still applies below 640px) so it is safe to add retroactively to existing wrap-based patterns.
- **Flex `justify-between` needs explicit `gap` for minimum spacing (STORY-26)**: When adjacent text/inline elements are spaced with `justify-between` (e.g., a date label and a state label in a toggle row), static CSS reasoning that "they won't overlap" is insufficient if content length varies. At 375px viewport with long pt-PT strings (e.g., "domingo, 6 de setembro de 2026" + "Disponível"), Tailwind's `justify-between` without `gap` leaves only ~1.9px actual spacing — visually merging adjacent text despite no technical overlap. QA caught this via real-browser rendering, which automated scrollWidth/tap-target checks did not. Fix: add `gap-2` (or appropriate Tailwind gap utility) to guarantee minimum inter-element breathing room regardless of content length. Pattern: `<div className="flex justify-between gap-2">`. This is especially critical for i18n'd layouts where translated string length is unpredictable. Verify via compiled CSS output and headless Chromium rendering at tight viewports, not visual reasoning alone.
- **Nested independent flex-wrap contexts produce incoherent layouts despite passing scrollWidth check (BUGFIX-06)**: A layout bug can escape regression detection when two independent flex containers nest — e.g., an outer wrapper with `flex flex-wrap` and an inner child with its own `flex flex-wrap` inside. If each satisfies `document.documentElement.scrollWidth <= viewport` independently, scrollWidth-only testing passes while the visual layout breaks (e.g., nav content wrapping above the logo, avatar dropping to a third row). The metric is insufficient because it only measures horizontal overflow, not coherence across multiple wrapping contexts. **Fix pattern**: flatten nested wrap contexts into a single, coordinated flex-wrap container. Example: `AppHeader.tsx`'s outer row (`flex flex-wrap items-center gap-2 sm:flex-nowrap`) with three direct children (logo, avatar div, nav div) — no nesting — so only one wrap decision point exists. At 375px, row 1 contains logo + avatar-pinned-right, row 2+ contains nav links wrapping across full width. The fix required reordering JSX (`Link → avatar-div → nav-div`) to match mobile's desired visual order without CSS `order` at the default breakpoint. When adding new interactive elements to a header/nav after a wrap-based fix ships, re-verify at both 375px and the real-device-width-from-the-bug-report (not just scrollWidth), and confirm via screenshot that wrapping looks intentional, not broken. A scrollWidth pass is necessary but not sufficient.
- **CSS `order` for breakpoint-specific visual reordering creates DOM/tab-order divergence (BUGFIX-06 accepted risk)**: When a component's desired visual order differs between breakpoints (e.g., header shows logo → nav → avatar on desktop but logo → avatar → nav on mobile), a single DOM order can match only one breakpoint's default (no-`order`) visual order. Using CSS `order` to override one breakpoint inevitably diverges DOM order from visual order at the other breakpoint — a keyboard user's tab order will not follow on-screen left-to-right. **Decision criterion**: pick the breakpoint that is the bug's actual subject (usually mobile/375px for responsive layout fixes) as the bug-free case with zero `order` (DOM == visual == tab order at that breakpoint). Document the accepted opposite-breakpoint divergence (e.g., desktop tab order inverted from on-screen order) in an inline code comment explaining it's deliberate, narrow, and why eliminating it (e.g., via HTML-level DOM duplication with per-breakpoint rendering) was rejected as too invasive. Verify both breakpoints manually — the bug-free breakpoint's tab order via keyboard walkthrough, and the accepted-divergence breakpoint's tab order to confirm it's present as documented, not a missed regression. This pattern is modeled on CLAUDE.md's STORY-18 TOCTOU-acceptance precedent: explicit, documented, narrow risk accepted at the lowest level (single component comment, single breakpoint, reversible). See BUGFIX-06's `AppHeader.tsx` comment (lines 52–79) and cycle-2 revision note for the worked example.
- **Closing "E2E_WITH_AUTH-gated tests never run in CI" gaps via e2e-integration with shared fixtures (BUGFIX-06 pattern)**: When a regression test exists but is gated behind `test.skip(!process.env.E2E_WITH_AUTH, ...)` (only runs for developers with real Google OAuth credentials), it provides zero CI coverage and regressions ship undetected. Supplementing/replacing such tests with a real-browser authenticated suite in `e2e-integration/*` using the established `adminPage`/`memberPage` fixtures (CHORE-15 precedent — real cookies via `signInWithPassword` against a local Supabase instance) closes the gap structurally: the new spec runs unconditionally in the `integration-test` CI job on every PR, no opt-in required. Pattern: create a new `e2e-integration/<concern>.spec.ts` (not appended to an existing spec, to keep concerns separated), model it on `home.spec.ts`'s existing `adminPage`/`memberPage` setup, and parametrize assertions over multiple widths/roles/pages as the story's AC requires (example: BUGFIX-06's `e2e-integration/header-nav-mobile-overflow.spec.ts` parametrizes over `[375, 390]` widths and `[admin, member]` roles to catch regressions on each addition of a nav link). If a screenshot or manual visual check is required (e.g., "visually confirm wrapping looks intentional, not broken"), capture it via `page.screenshot()` as a CI artifact and document in Manual Verification / PR description that the human review must explicitly check for the specific pattern (e.g., "confirm the 4th nav link wraps alone on the right, reads as intentional"). This avoids the false confidence of "we have an automated test" when the test's output artifact actually requires human judgment — document that expectation up front.
- **shadcn Button with `asChild` role shift in tests (CHORE-10)**: When a Button is rendered with the `asChild` prop (e.g., `<Button asChild><Link>...</Link></Button>`), the button's accessible role becomes the role of its child element, not `button`. A `<Link>` child renders as `link`, not `button`. When writing tests for such components (e.g., `AppNav.tsx` renders nav links via `<Button asChild><Link>`), use `getByRole('link')` instead of `getByRole('button')`. Additionally, always extract button text from the `messages/pt-PT.json` source file rather than guessing or inferring it — a test using fabricated text (e.g., `/Ativar|Disabled/`) will fail immediately for any developer running it locally and provide zero AC coverage despite appearing complete. Pattern: find the key in the component source (`Nav.userManagement` in AppNav.tsx), then verify the value in `messages/pt-PT.json` ("Utilizadores"), then use that exact string in the test locator: `page.getByRole('link', { name: 'Utilizadores' })`. For button-text extraction, always check the translation file first — this discipline catches test bugs before they block story delivery. See `e2e/button-cursor.spec.ts` lines 44–48 (admin nav link) and lines 57–62 (home CTA button) for reference.
- **E2E test count assertions on nav links (STORY-17)**: When a story adds a new admin nav link via `AppNav.tsx`, the regression test `e2e/app-nav.spec.ts` has an exact-count assertion (e.g., `await expect(nav.getByRole('link')).toHaveCount(2)`) that must be updated in the same story. Additionally, any e2e spec file with a **doc comment stating a count** (e.g., "AC3: Admin nav shows exactly 2 links") must have that count verified and updated in the same step that adds the UI element — doc comments drift silently without test failures unless explicitly updated. Checklist: (1) Increment the count in `toHaveCount(N)`, (2) add an assertion for the new link's text (extract from `messages/pt-PT.json`), (3) update the test title and file header comments to reflect the new count and list all links by name.
- **Test-docstring discipline for decorative-only visual elements (CHORE-27)**: When a test asserts the presence of a decorative visual element (e.g., a border, ring, shadow) that is intentionally not contrast-gated to WCAG standards, the test's docstring or title must explicitly state it is decorative/visual-separation-only — not WCAG-verified. Bare test presence without this labeling can mislead future readers into assuming test coverage implies accessibility compliance. Example: a test asserting `dark:ring-1 dark:ring-header-border` presence should document "decorative border-only (1.69:1 contrast, below 3:1 floor)" so reviewers understand the ring is visual separation, not a contrast-meeting element. This discipline prevents silent misunderstandings about what test coverage actually proves and is particularly important for component-level tests that touch color tokens and global elements.
- **Rich-text tag introduction breaking raw-string test comparisons (CHORE-25)**: When adding `<num>...</num>` or other rich-text tags to an already-shipped i18n key, audit every raw-string comparison site in e2e specs that builds expected values via `.replace()` or similar template substitution. The tags are consumed during `t.rich()` rendering (replaced by render-prop output), but test helpers doing their own string substitution still produce literal `<num>` text in their expected values, causing `toContainText` failures. Fix: strip tags before substitution: `messages.Home.keyName.replace(/<\/?num>/g, '').replace('{placeholder}', value)`. Mitigate: grep all e2e files for `.replace(` call sites whenever updating an i18n key template, cross-checking each against the affected key's new tags. Do not assume shared helpers like `renderPlural()` cover all sites — any direct raw-template comparison must be updated too. Example: `e2e-integration/home.spec.ts` (CHORE-25) required identical tag-stripping fixes in both the shared `renderPlural()` helper and the direct `expectedNextBlockedText` computation.

The first CI run after manually-applied migrations (e.g., via SQL Editor) will attempt to re-apply them, which is safe only if SQL uses guards (`IF NOT EXISTS`, `DROP POLICY IF EXISTS CREATE POLICY`; PG15 compatible).

CI jobs should always pass `--project-ref` explicitly so they target the correct environment regardless of config.toml. The config.toml can safely use a placeholder dev ref locally.

A failing `migrate` job does NOT cancel an in-flight Vercel build. To close this gap, enable "Require CI to pass before deploying" in Vercel project settings → GitHub integration (manual operator step). This prevents future gaps when service-role operations are added; the placeholder allows builds to succeed without real credentials.
- **`supabase status -o env` produces quoted values (CHORE-05)**: The `supabase status -o env` command wraps each value in literal double quotes (e.g. `ANON_KEY="eyJhbGci..."`). Piping this directly into `$GITHUB_ENV` does no unquoting — the env var ends up with leading/trailing `"` characters, corrupting JWT tokens and breaking Supabase client calls. **Next.js 16 Route Handler dynamic params with auth guards:**
- When a dynamic Route Handler has both an auth guard AND `{ params }: { params: Promise<...> }`, always call the guard FIRST before awaiting params. The guard can short-circuit with 401/403 before params are resolved. Even though `await params` is cheap (URL parsing), this ordering is canonical: `const result = await requireAdmin(request); if (result instanceof NextResponse) return result; const { id } = await params; ...`

**Route Handler input validation (JSON body + path params):**
- **Invalid JSON body → 400, not 500**: Wrap `request.json()` in its own `try/catch` BEFORE the outer DB try block. A SyntaxError from malformed JSON must return HTTP 400, not 500. Pattern: `let body; try { body = await request.json() } catch { return NextResponse.json({ error: 'invalid_json' }, { status: 400 }) }`
- **UUID path param validation → 400, not 500**: Dynamic Route Handlers must validate UUID format before querying Supabase. PostgreSQL error `22P02` (invalid_text_representation) surfaces as 500 without this guard. Use `const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i; if (!UUID_RE.test(id)) return NextResponse.json({ error: 'invalid_id' }, { status: 400 })`
- **UTC-safe day-of-week validation** (STORY-25): When validating a date string is both a real calendar date AND a specific weekday (e.g., a Sunday), use regex + UTC round-trip math, **never** `new Date(dateStr)` directly (which parses in local/host timezone and could shift the effective weekday). Pattern: regex `^\d{4}-\d{2}-\d{2}$`, then `Date.UTC(y, m-1, d)` and verify `getUTCFullYear/getUTCMonth/getUTCDate` match the parsed input (catches rollover dates like `2026-02-30`). Keep date validation and weekday validation in separate functions so Route Handlers can emit distinct error codes (e.g., `invalid_date` vs `not_sunday`). Example: `lib/validation/availability.ts` (STORY-25) exports `parseBlockedDate()` for format/existence and `isSunday()` for weekday check, both using UTC construction.
- **Case-insensitive UUID comparison for auth checks** (STORY-08): When comparing a path-param user id against an auth-derived id (e.g., `userId === result.user.id`), always call `.toLowerCase()` on both sides before the equality check. Although Supabase generates lowercase UUIDs consistently, a path-param string supplied by a caller should not be trusted to have matching case. A case mismatch silently fails the JS equality check while a downstream `.eq('id', userId)` Postgres query still matches the same row (Postgres normalizes UUID case, JS string equality doesn't) — a bypass where the auth check fails but the DB write still succeeds. Pattern: `if (userId.toLowerCase() === result.user.id.toLowerCase() && role === 'member') { return NextResponse.json({ error: 'self_demotion' }, { status: 400 }) }`.
- **Stable machine-readable error codes** (STORY-17): When a Route Handler must distinguish between multiple failure modes (e.g., blank name, invalid slot count, duplicate name), use stable, machine-readable error codes (not free-text English strings) in the error response: `{ error: 'name_required', status: 400 }`, `{ error: 'invalid_slots', status: 400 }`, `{ error: 'duplicate_name', status: 409 }`, etc. This allows the client (Client Component or e2e test) to map specific codes to specific i18n messages or assertions. **HTTP status semantics**: 400 for malformed/invalid input (name blank, slot count ≤ 0, non-numeric, bad JSON, bad UUID), 409 for state conflict (Postgres 23505 unique violation → duplicate key). Existing precedent: `last_admin` 409 in `app/api/admin/users/[id]/route.ts`. Client error-mapping example: `const errorMap = { 'name_required': t('RoleManagement.errorNameRequired'), 'invalid_slots': t('RoleManagement.errorInvalidSlots'), ... }; const message = errorMap[code] ?? t('RoleManagement.errorGeneric')`.
- **Field validation helper module**: When a Route Handler needs a validation helper (e.g., parsing a field's value to distinguish omitted/default from invalid), place it in `lib/validation/<domain>.ts` (not co-located in a route file) so multiple route files can share it without creating circular imports or sibling-file dependencies. Pattern: `lib/validation/roles.ts` exports `parseDefaultSlots(raw: unknown): number | null`, where `undefined/null → 1` (default, AC2) and any invalid value (blank, ≤0, non-numeric) → `null` (to trigger 400, AC3). Clients import with `import { parseDefaultSlots } from '@/lib/validation/roles'` (STORY-17).

**Shared types between Server and Client Components:**
- When an interface is used in both a Server Component and a `'use client'` Client Component, place it in `types/<domain>.ts` with zero server dependencies (no `import 'server-only'`; no server-only imports). Both components import with `import type { Interface } from '@/types/<domain>'`. This avoids `server-only` module contamination if the type were co-located with server code. Example: `types/user-management.ts` exports `UserRow` used by both `app/[locale]/admin/users/page.tsx` and `components/UserTable.tsx`.

**Supabase SSR Auth (Next.js 16):**
- **Packages**: Install `@supabase/supabase-js` and `@supabase/ssr`.
- **Server client** (`lib/supabase/server.ts`): Use `createServerClient` from `@supabase/ssr` with `next/headers` cookies. The `setAll` method must be wrapped in try/catch because Server Components have read-only cookie stores; calling `setAll` in a Server Component throws. The proxy.ts (middleware) path never calls `setAll` — only Route Handlers (writable) do.
- **Route Handler cookie gotcha (Next.js 16)**: `await cookies()` from `next/headers` does NOT reliably forward the browser's cookies to Route Handlers — `auth.getUser()` returns null with no outgoing request despite cookies being present in the browser. Use `request.cookies.getAll()` directly (the same approach as proxy.ts). Auth guard functions and any Route Handler that reads the session MUST accept a `NextRequest` parameter and read `request.cookies`. Do not use `createClient()` from `lib/supabase/server.ts` inside Route Handlers; construct the Supabase client directly with `request.cookies`.
- **Route Handler `setAll` must write to `request.cookies`**: With `setAll` as a no-op, an expired access token causes `auth.getUser()` to fail even after a successful server-side refresh — the Supabase client's in-memory session is not updated if `setAll` does nothing. In the guard's `setAll`, call `request.cookies.set(name, value)` for each refreshed cookie. This updates the in-memory request object so the client can use the new access token for the current request. The browser receives updated tokens on the next page navigation via proxy.ts.
- **Server-side auth guard pattern** (`lib/auth/guard.ts`): Export `requireAuth(request: NextRequest)` and `requireAdmin(request: NextRequest)`. Both return `AuthUser | NextResponse`. Callers check `if (result instanceof NextResponse) return result`. The guard reads cookies from `request.cookies` (not `next/headers`). Future Admin-only Route Handlers follow the same pattern: `const result = await requireAdmin(request); if (result instanceof NextResponse) return result; ...`.
- **Runtime role validation**: Never use bare `as 'admin' | 'member'` casts on DB values. Use a ternary: `row.role === 'admin' ? ('admin' as const) : ('member' as const)`. A DB value that bypassed the CHECK constraint otherwise silently passes into the type system.
- **`console.error` in Route Handler catch blocks**: Always bind `catch (err)` and log `console.error('[guardName] unexpected error:', err)`. Route Handlers are a better log site than middleware — they run per-request with user context.
- **Supabase client error field in Server Components and Route Handlers**: The Supabase JS client does NOT throw on query errors — it returns `{ data: null, error: {...} }`. Always destructure `{ data, error }` and log `console.error` when `error` is truthy **before** treating it as success. A bare try/catch only catches JS exceptions (network failure, SyntaxError). Without checking the `error` field, silently returns empty data when the query fails. Pattern: `const { data, error } = await client.from('table').select(...); if (error) { console.error('[fnName] DB error:', error); ... return []; }`. This applies to all Supabase queries in Server Components, Route Handlers, and async helpers like `session.ts`. **Existing gap**: `app/[locale]/admin/users/page.tsx` has this bug (line 47 only destructures `{ data }`, not `{ data, error }`). Fix it in the next story that touches that file.
- **Browser client** (`lib/supabase/client.ts`): Use `createBrowserClient` from `@supabase/ssr`; synchronous factory for Client Components. Requires `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`.
- **OAuth initiation**: Call `signInWithOAuth` from the browser client with `provider: 'google'` and `redirectTo: window.location.origin + '/auth/callback'`. This avoids needing a `NEXT_PUBLIC_SITE_URL` env var. Always destructure `{ error }` synchronously and render an alert if present — the call does not redirect to the browser if the provider is misconfigured.
- **Multi-environment OAuth**: A single OAuth client (e.g., Google Client ID/Secret) can have multiple redirect URIs, allowing it to be reused across dev and prod Supabase projects. Add both `http://localhost:3000/auth/callback` (dev) and `https://yourdomain.com/auth/callback` (prod) as authorized redirect URIs in Google Cloud Console; both Supabase projects reference the same OAuth client.
- **Session refresh in proxy.ts**: Wrap the **entire** Supabase block (client construction + `getUser()`) in a single try/catch. Any error (missing env, network failure, invalid creds) falls through as `user = null` rather than throwing a 500. Copy `Set-Cookie` headers from the Supabase response onto the outgoing next-intl response.
- **Bidirectional auth guards in proxy.ts**: When adding guards, consider BOTH directions: `!user && !isLoginPath` (unauthenticated users on protected routes) AND `user && isLoginPath` (authenticated users on login). Both branches must copy Supabase cookies onto the redirect response (STORY-09).
- **Sign-out race condition and marker-cookie fix** (STORY-15): `@supabase/auth-js`'s `_signOut()` never clears the session cookie until after its network revoke call completes — there is no synchronous local clear, so client code cannot "race" the network with a timeout. A "sign out then navigate to /login" flow will hit proxy.ts's reverse-guard (`if (user && isLoginPath)`) bounce-back whenever the real signOut network call is still in flight. The structural fix: have the client synchronously set a short-lived, first-party marker cookie (e.g. `app-signout-pending=1`, `max-age=15`) via `document.cookie` **before** navigating (zero network, zero wait), then have proxy.ts read this marker and skip the Supabase session lookup entirely, treating `user` as `null` for that request. The marker only ever forces the stricter (logged-out) branch of either guard, never grants access, so it introduces no privilege-escalation risk. Extract the marker cookie name and max-age into a shared constants module (e.g. `lib/auth/signout-marker.ts`) imported by all consumers (middleware, client component, Route Handler). The marker must be cleared on every response path in the OAuth callback route (`app/auth/callback/route.ts`), so a sign-out-then-immediate-re-login isn't shadowed by a stale marker. See proxy.ts, components/UserWidgetMenu.tsx, and app/auth/callback/route.ts for the implementation pattern.
- **Auth callback return-path invariant** (STORY-11): Every return or redirect in `app/auth/callback/route.ts` **must** go through the `clearSignoutMarker(...)` helper. This is load-bearing for STORY-15's sign-out marker fix — any new redirect path added to the callback (e.g., redirecting a new user to `/claim` instead of home) must wrap that redirect through `clearSignoutMarker()` without exception. Pattern: compute the redirect target conditionally into a single variable across all branches, then apply a single `return clearSignoutMarker(NextResponse.redirect(...))` at the end, rather than adding parallel new return statements per branch. When adding new conditional logic to the callback, always check the doc comment at the top of the file and re-verify that every return path still goes through `clearSignoutMarker()`. See STORY-11's implementation (app/auth/callback/route.ts, lines 198-221) for the worked example: the new `/claim` redirect is computed conditionally into `redirectTarget`, then returned once through `clearSignoutMarker()` alongside the home redirect.
- **Auth guard regex**: Match protected paths with `/^\/[^/]+\/(login|auth)(\/|$)/` or segment anchors, **not** substring matchers like `\/login/`. Substrings incorrectly match `/pt-PT/login-admin` and bypass the guard. The matcher should exclude `/auth/*` from next-intl processing so OAuth callbacks are never locale-prefixed.
- **Middleware matcher comment**: Add an explicit comment near `config.matcher` noting that `/auth/*` exclusion makes any `isAuthCallback` checks inside the middleware function dependencies — if a future dev removes the exclusion, those checks become dead code. This prevents silent bugs.
- **searchParams Promise**: In Next.js 16, page props have `searchParams: Promise<object>`. Type it as `Promise<{ error?: string }>` and `await` before reading properties.
- **signOut server action**: Wrap `createClient()` + `signOut()` in try/catch. Always call `redirect()` unconditionally after the catch block, so the user reaches login regardless of errors. Use `routing.defaultLocale` (from `@/i18n/routing`) for the redirect path, not hardcoded strings.
- **User display fallback**: Optional user metadata fields (e.g. `user.user_metadata.full_name`) may be absent. Always provide a non-empty fallback when rendering user greetings (e.g. via an i18n key like `Auth.userFallback` that covers "Olá, " with a default name) so the UI never shows truncated text like "Olá, ".
- **Access-denied banner guard in Server Components**: When a query param (e.g. `?denied=1`) drives a UI denial message, always guard it with `role !== <privileged_role>`. Without the guard, a privileged user who bookmarks or crafts a `?denied=1` URL sees a misleading "you don't have permission" message. Pattern: `const showDeniedBanner = denied === '1' && role !== 'admin'`. Apply `showDeniedBanner` to all code paths that render user-facing content, not just the denial branch.

**React cache() for auth deduplication:**
- Create `lib/auth/session.ts` with `getSessionUser` and `getUserRole(userId)` wrapped in `cache()` from `'react'`. These are the canonical helpers for all Server Components needing the current user or role.
- **Verify cache deduplication when adding layout-wide auth-derived values (CHORE-22)**: When a story adds a new auth-derived value (e.g., `role`) to a layout that wraps every authenticated page, always check which `cache()`-wrapped helpers are already called on **every** page, not just some. For example, when adding role-fetch to `app/[locale]/(app)/layout.tsx`, verifying that `AppHeader` already calls `getUserProfile(user.id)` with the same argument on every route guarantees zero extra round-trips — but if the layout had instead called a different helper like `getUserRole()`, routes like `settings/page.tsx` (which call `getUserProfile`, not `getUserRole`) would incur a genuine second DB query, only visible if the whole route tree is audited. The cache dedupes per *helper function*, not per route — identical arguments to the same `cache()`-wrapped helper call collapse into one round-trip per request. Mistake: skipping the audit and assuming "we only call this on the routes that already fetch auth" leads to silent N+1 queries on routes you didn't think about.
- **`server-only` marker omission for standalone generic-parameter modules** (STORY-25): Omit `import 'server-only'` in two cases: (1) **Transitive server boundary** — `next/headers` or similar is imported transitively via dependencies (e.g., `lib/auth/session.ts`), giving the module a bundler-level server boundary without the explicit marker. (2) **Standalone with no transitive boundary** — the module takes only generic parameters and embeds no secrets, AND it must be directly importable by Node-based Playwright integration tests (`e2e-integration/*.spec.ts`) to exercise query filtering or helper logic on seeded fixture data. `server-only`'s package.json only swaps to the safe empty export under a `react-server` condition set by bundlers (Next.js, webpack), so adding the marker causes a synchronous throw when imported directly by Playwright's plain Node test runner. Example: `lib/availability/blocked-dates.ts` (STORY-25) exports `getBlockedDates(client, opts)` — no transitive boundary, no secrets, but AC8's test plan requires direct import to verify `personIds`/`dateRange` filtering. Omit the marker in this case. (3) **Do NOT omit otherwise** — service-role client factories and auth helpers still need the marker despite module-level visibility. Apply this pattern narrowly; when in doubt, include the marker.
- `React.cache()` scopes memoisation to a single server render tree — there is no cross-request cache leak risk.
- `supabase.auth.getUser()` returns `{ data: { user: User | null } }`. The `user` field is `User | null`, never `undefined`. Do not add `?? null` — it is dead code.
- All catch blocks must bind `catch (err)` and call `console.error('[fnName] unexpected error:', err)` before returning null. Bare `catch {}` is not acceptable in server helpers.
- The middleware (`proxy.ts`) runs in a different execution context and cannot share the React cache — it must keep its own `auth.getUser()` call.

**Client-side sort order (Optimistic updates in Client Components):**
- When a Server Component fetches data with `.order('name', { ascending: true })` and passes it to a `'use client'` Client Component, any optimistic state update (add/edit/remove) that inserts or modifies rows must maintain the server's sort order. After a successful API call, re-sort the state array using the same criteria: `setRows(prev => [...prev, newRow].sort((a, b) => a.name.localeCompare(b.name, 'pt')))`. Failing to re-sort breaks the UI sort order until the next page reload.

**Keyed in-flight state for multi-item lists (STORY-18):**
- When rendering a list of independently-actionable items (e.g., roles with async save-on-tap buttons) and tracking which item's request is currently in-flight, use a keyed structure (`Set<string>` or `Record<string, boolean>`) instead of a scalar state (`inFlightId: string | null`). A scalar cannot distinguish between "no request in flight", "request A in flight", and "concurrent requests A and B in flight" — when item B's save starts while item A's is still pending, the scalar assignment `setInFlightId(B)` will clobber A's state, removing the disabled gate on A's buttons before A's request resolves. Pattern: `const [savingRoleIds, setSavingRoleIds] = useState<Set<string>>(() => new Set())`, then gate with `disabled={savingRoleIds.has(roleId)}`, add on request with `setSavingRoleIds(prev => new Set(prev).add(roleId))`, and delete in finally with `setSavingRoleIds(prev => { const next = new Set(prev); next.delete(roleId); return next; })`. This ensures each item's request is independently tracked and concurrent operations don't interfere. See `components/PersonSkillsEditor.tsx` (STORY-18) for the worked example.

**Extending a Member-facing Client Component for admin use via optional props (STORY-27):**
- When a `'use client'` interactive component (e.g., a toggle list, a form) must serve both a Member self-service path and an admin-on-behalf path, avoid duplicating it. Instead, add optional props that gate only the URLs, title text, and optional UI elements, leaving the state machine (useState, side effects, event handlers) untouched. Pattern: `AvailabilityToggleList` receives optional `personId?: string` and `personName?: string` props that default to `undefined`. `const isAdminMode = personId !== undefined` gates the endpoint URLs (`/api/admin/people/${personId}/availability` vs `/api/availability/blocks`), the page title, and a "back to team" link. The rest — `blockedDates` / `pendingDates` / `errorMessage` state, optimistic flip/revert logic, `finally` cleanup — is untouched, so the already-shipped Member path (which never passes `personId`) has zero behavioral change. This pattern scales to other shared interactive components. See `components/AvailabilityToggleList.tsx` (STORY-27, lines 38–67) for the reference. **Regression testing**: always re-run the Member feature's e2e suite (e.g., `e2e-integration/availability.spec.ts`) unmodified after the component edit to prove the Member path still works.

**Per-page admin guard convention (Server Components):**
- Every `app/[locale]/admin/*/page.tsx` must include a role-guard redirect until a middleware-level guard is introduced. This is belt-and-suspenders: `proxy.ts` guards unauthenticated access, but members who somehow reach an admin URL need per-page rejection.
- Pattern: Fetch the user's role, check `if (role !== 'admin') redirect(\`/${routing.defaultLocale}/?denied=1\`)`. The `?denied=1` query param signals the home page to show an access-denied banner (use the `showDeniedBanner` guard pattern documented in Supabase SSR Auth).
- This convention is per-page opt-in — there is no structural enforcement at the framework level in the current architecture (see ADR-007 for the rationale). As the admin section grows, migrate to a middleware-level guard if page duplication becomes burdensome.

**Admin-on-behalf Route Handler pattern (STORY-27):**
- When a Route Handler must allow an admin to perform an action on behalf of another user/person (e.g., blocking a person's availability, assigning a person to a role), mirror the existing self-service Route Handler's structure but swap `resolveSelfPersonId`-based resolution for direct `person_id` from the URL param, with `requireAdmin` guarding instead of `requireAuth`. Pattern: (1) `requireAdmin` guard first, then `await params` (CLAUDE.md: Route Handler dynamic params with auth guards); (2) UUID validation on `id` → 400 `invalid_id`; (3) person lookup with `is_active = true` check → 404 `not_found`; (4) all validation (date format, business rules, etc.) before any writes, just as in self-service routes; (5) idempotent upsert/delete operations matching the self-service semantics exactly. The effect: one shared `blocked_dates` table, one shared state machine in the Client Component (e.g., `AvailabilityToggleList`), no admin-specific storage or schema. Example: `app/api/admin/people/[id]/availability/route.ts` (STORY-27 GET+POST) and `[date]/route.ts` (DELETE) mirror `app/api/availability/blocks/route.ts` (STORY-25 GET+POST) and `[date]/route.ts` exactly in shape, differing only in the person-id source and auth guard. This pattern is reusable for any "admin does Y for person X" feature (EPIC-05 admin operations, etc.). Each handler should include a comment documenting its role-mirroring (see STORY-27 GET handler line 10–22 for the reference pattern).

**Error-code consistency across admin-on-behalf routes:**
- The sibling admin Route Handlers in `app/api/admin/people/[id]/*` have evolved to use `not_found` for person-not-found cases (GET availability route, GET skills route), with an earlier PUT skills route using `person_not_found`. When adding future admin-on-behalf routes, prefer `not_found` for consistency with the GET handlers' 404 responses, rather than inventing new spellings like `invalid_person_id` or `person_not_found`. Consistent error codes reduce test-author confusion and prevent cascading rework. An explicit comment in each new handler (pointing back to this note) helps future developers understand why the code-string was chosen over other candidates. See STORY-27 Design decision 4 post-review update for the rationale behind this choice.

**Soft-delete + partial unique index** (STORY-17): When a table uses soft-delete (`is_active BOOLEAN DEFAULT true`) and needs a unique constraint on a field (e.g., role name), use a partial unique index instead of a table-level UNIQUE constraint: `CREATE UNIQUE INDEX idx_roles_active_name_lower ON public.roles (lower(name)) WHERE is_active`. This allows soft-deleted rows' values to be reused (since the index only covers `is_active = true` rows) while maintaining uniqueness for active rows. Deletion becomes `UPDATE roles SET is_active = false WHERE id = ?` (soft-delete), not a hard DELETE. Combine with `INSERT ... SELECT ... WHERE is_active = true` queries to only see active rows. Example: users can delete a role named "Sound", then create a new role also named "Sound" — the old one is invisible (soft-deleted) and doesn't block the new one.

**Soft-delete does NOT trigger ON DELETE CASCADE (STORY-19 critical correction)**: When a table has `ON DELETE CASCADE` foreign keys pointing to its children (e.g., `person_role_skills` has FK to `roles` with CASCADE), those cascades **only fire on actual hard DELETE statements, never on UPDATE**. If a Route Handler soft-deletes the parent row (e.g., `UPDATE roles SET is_active = false WHERE id = ?`), PostgreSQL never touches the child rows — the CASCADE sits dormant. If the design requires physically removing child rows (not just filtering them as invisible), the Route Handler must issue an explicit hard `DELETE FROM child_table WHERE parent_id = ?` **before** the soft-delete UPDATE. Order matters: delete children first, then soft-delete parent. If the child-delete fails, the parent remains `is_active = true` and the error is visible to the user, allowing retry. If the parent-delete fails after children are gone, the parent still appears in the list but its data is lost — accepted as a narrow TOCTOU risk for low-concurrency admin surfaces (document this explicitly as a code comment). Pattern: `const { error: childError } = await serviceClient.from('person_role_skills').delete().eq('role_id', roleId); if (childError) { console.error('[deleteRole] child delete error:', childError); return NextResponse.json({ error: 'internal' }, { status: 500 }); } await serviceClient.from('roles').update({ is_active: false }).eq('id', roleId);`. This pattern applies to any Route Handler soft-deleting a parent with dependent children — verify at code-review time whether the cascade is happening implicitly (dangerous, silent data loss) or explicitly (deliberate, recoverable).

**Stale-link exclusion pool when partial unique index lacks is_active filter** (STORY-20): When a table has a soft-delete column and a partial unique index that covers only non-null values (e.g. `CREATE UNIQUE INDEX idx_people_linked_user_id_unique ON people (linked_user_id) WHERE linked_user_id IS NOT NULL` from STORY-11), a soft-deleted row still occupies its linked-item slot at the DB level. If a picker UI filters the exclusion pool to only *active* rows, it will offer items that look available but will be rejected by the unique constraint (e.g., "this user is already linked" with no visible person holding the old link). Fix: query the exclusion pool **unfiltered** to include soft-deleted rows. Pattern: in `app/[locale]/admin/people/page.tsx`, run `SELECT linked_user_id FROM people WHERE linked_user_id IS NOT NULL` (no `is_active` filter) to compute the true "taken" set, then pass it to the picker component. The picker filters against this unfiltered set, avoiding false-availability confusions. This applies to any future feature with a picker UI + soft-delete + partial-unique-without-is_active pattern. See STORY-20 Design decision 5 for context.

**Metric/count consistency across same-UI filters** (STORY-30): When rendering multiple metrics/counts on the same page or card (e.g., "N active people" + "N active roles" + "N people with recent blocks"), all counts must be derived using identical scope filters — never derive one count's filter separately from another's. A data-integrity trap occurs when mixing scoped and unscoped queries: e.g., active-people count from `people WHERE is_active=true` looks consistent with a blocks-count from `blocked_dates` (unfiltered), but the second count includes stale blocks from soft-deleted people (since soft-delete never cascades to blocked_dates rows per STORY-19), creating a misleading metric that contradicts the active-people filtering next to it. Fix: compute the active-person ids once (`people.select('id').eq('is_active', true)`), derive the active-people count from this list's length, and pass the same list as the `personIds` filter to any blocked-dates aggregation query. The result: both counts reflect the same active-person scope, and the metrics stay semantically aligned. This applies whenever dashboard-like UIs show side-by-side scope-dependent metrics — audit at review time that all filters agree. Example: `app/[locale]/(app)/page.tsx` (STORY-30) AC3 and AC4 branches.

**Data-model-aware neutral UI copy framing** (STORY-30): When a feature's data model has an "absence-of-row means default-positive-state" design (e.g., `blocked_dates` rows only exist for explicitly-blocked dates; no row = fully available), any user-facing count or summary must frame the data neutrally without implying absence is an incomplete/warning state. Naive copy like "N people haven't set their availability" or "availability pending" misrepresents the model to users. Correct framing uses factual, neutral language: "N people with blocks recorded in the next 30 days" (a count of rows that exist, not an absence-of-rows warning). To prevent silent copy regressions (e.g., a future edit adding a warning-toned substring like "pendente" or "pending"), add a CI-safe static check to `e2e/i18n-neutral-copy.spec.ts` (reads locale JSON files directly, searches for forbidden substrings in guarded keys, runs at lint time). When shipping a feature with this data-model constraint, document the guarded keys explicitly in the test file and in code comments (see `app/[locale]/(app)/page.tsx` lines 183–184 and 245–253 for examples). This pattern is reusable for any future feature (e.g., scheduling data where absence-of-assignment means "not yet scheduled, a valid state" not "incomplete").

**PostgREST soft-delete join filter** (STORY-18): When querying a join table with soft-deleted parent rows and only active parents are meaningful (e.g., person-role skills where roles can be soft-deleted), use `select('...join, parent!inner(is_active)')` + `.eq('parent.is_active', true)`. The `!inner` forces an INNER JOIN against the parent table, and `.eq('parent.is_active', true)` filters on the joined column server-side. This makes "only active-parent rows qualify" an explicit part of the query, not a comment-only contract that callers must remember. A caller cannot forget the filter because it's baked into the one query the helper exposes. This is the single source of truth for "qualified set" queries that EPIC-04 and other features will depend on. Pattern: `lib/skills/qualified-roles.ts` (STORY-18) demonstrates this for the `person_role_skills` join to `roles`.

**TOCTOU acceptance in admin-only endpoints (STORY-18):**
- Certain admin-only Route Handlers may have narrow time-of-check-to-time-of-use (TOCTOU) race windows where a checked precondition (e.g., a role is active) could change between the check and the subsequent write. Preventing this requires database transactions or locks, which add complexity and latency. If the race condition is of low severity (e.g., silently assigning a skill to a soft-deleted role defeats the "only active roles" guard, but the skill row is invisible anyway since the role is inactive) and the actual likelihood is vanishingly small (admin-only surface, few concurrent users), explicitly document the accepted risk as a comment in the handler rather than adding machinery to prevent it. Pattern: comment lines 91-96 in `app/api/admin/people/[id]/skills/[roleId]/route.ts` (STORY-18) documents the accepted TOCTOU between the role active-check and the upsert. Do not apply this pattern lightly; it is an explicit exception to "always prevent race conditions," not a default.

**GRANT before RLS (PostgreSQL prerequisite):**
- PostgreSQL checks table-level privileges BEFORE evaluating RLS policies. If `authenticated` (or `anon`) has no SELECT privilege on a table, every query returns `42501 permission denied` — RLS policies are never consulted. Always include `GRANT SELECT ON public.<table> TO authenticated;` (and other verbs as needed) in the same migration that creates the table and enables RLS. Symptom: Supabase client returns `{ code: '42501', message: 'permission denied for table <name>' }` even though the row exists and the user is authenticated.
- **`service_role` also needs explicit GRANTs**: The service-role client (used in admin Route Handlers) connects as `service_role`, not `authenticated`. PostgreSQL still checks table-level privileges for `service_role` even though it bypasses RLS. Every migration that creates a table used by Route Handlers must include **both**: `GRANT SELECT, INSERT, UPDATE, DELETE ON public.<table> TO authenticated;` AND `GRANT SELECT, INSERT, UPDATE, DELETE ON public.<table> TO service_role;` **in the same migration**. Missing the `service_role` grant causes `42501` in Route Handlers while the page (which reads via the same client but hits the anon path) silently returns an empty list. **Enforcement**: When a new Route Handler is introduced that writes to a table, the GRANT migration must land in the **same PR** as the Route Handler code. If the migration is deferred to a later story, the feature works in dev/prod with live Supabase projects (which may already have manual grants) but fails on fresh project re-provisioning from migrations. Prevent this by reviewing PR diffs: for any Route Handler using `.update()`, `.insert()`, or `.delete()`, verify the corresponding migrations grant `service_role` access. This project has hit this gap twice (STORY-03 `public.users`, STORY-14 `public.people`); STORY-17 avoided it by including both GRANTs in one migration (STORY-07 needed a follow-up fix-up migration) — catch it in code review before landing.
- **Service-role routes prove GRANT enforcement, not RLS enforcement (CHORE-14)**: When testing admin routes that use `createServiceClient()` internally for data reads, those reads bypass RLS policies (service-role ignores all RLS). A test asserting "200 with real data" from such a route proves GRANT table-level permissions but NOT RLS policy enforcement. The actual RLS proof comes from testing a non-admin user's rejection on the same route (expect 403), which exercises the auth guard's `auth.getUser()` read via the anon-key client (RLS-gated). When designing admin-route integration tests, cover both code paths: (1) admin GET succeeds with data → proves GRANTs in place; (2) member GET fails with 403 → proves auth guard + RLS together enforce access control. The distinction matters because they're different Supabase clients (service-role vs. anon-key) with different policy applicability. Example: `e2e-integration/admin-crud.spec.ts` (CHORE-14) tests AC1 (admin 200) and AC2 (member 403) to cover both paths.

**Supabase service-role client:**
- **Service-role client pattern**: Use `createClient` from `@supabase/supabase-js` (not `@supabase/ssr`; no cookie management needed). Add `import 'server-only'` as the first line of `lib/supabase/service.ts` to prevent accidental browser imports. Create the client lazily inside the factory function (not at module level) so missing env vars (e.g. `SUPABASE_SERVICE_ROLE_KEY`) don't throw at module-load time in CI builds with placeholder credentials.
- **Fail-closed host guard for privileged seed/bootstrap scripts (CHORE-05)**: Any script that uses the service-role client to create privileged test accounts (fixed UUIDs, committed credentials) or bootstrap admin accounts must validate its target Supabase URL is a local/ephemeral instance **before** constructing any Supabase client or making any network call. Use `new URL(url).hostname` against an exact allow-list (`127.0.0.1`, `localhost`), never a regex — regexes are bypassable (e.g., `http://127.0.0.1:8000@attacker.com` matches `^https?:\/\/(127\.0\.0\.1|localhost)(:|\/|$)` but connects to `attacker.com` via URL userinfo). The guard must run first, before client construction, with a clear error message if the check fails. This prevents a misconfigured `.env.local` from silently creating a privileged account against the shared dev or prod Supabase project. Add an explanatory comment in the script and in `supabase/config.toml` (near `project_id`) clarifying that seeding is CI-only infra, unrelated to the documented dev setup (e.g., CHORE-03's decision to use a shared cloud project for local dev, not Docker Supabase). Pattern: guard at script entry, `throw process.exit(1)` on failure.
- **Hoisting client-factory calls: re-check exception boundaries** (STORY-11): When refactoring control flow to hoist a client-factory call earlier (e.g., `const serviceClient = createServiceClient()` moved outside a try/catch), always re-verify what exceptions the factory can throw and **wrap the hoisted call in its own try/catch if needed**. The factory documentation may claim to be "synchronous and non-throwing," but the underlying `@supabase/supabase-js` `createClient()` can throw synchronously on missing or malformed env vars. If the call previously lived inside a try/catch that logged errors and fell through gracefully, hoisting it without a guard changes that behavior — the exception now propagates to an outer handler, which may produce a different (worse) user-facing result. Pattern: hoist the call, wrap it in a dedicated try/catch, set a sentinel (e.g., `let serviceClient = null`), and check the sentinel before using it downstream. Example: `app/auth/callback/route.ts` lines 170-175 wraps the hoisted `createServiceClient()` to preserve degraded-mode behavior if env vars are missing. This is especially critical in auth paths where exceptions can block login.
- **`CREATE POLICY IF NOT EXISTS` is PG17+ only**: Supabase runs PostgreSQL 17 (as of mid-2025). However, `DROP POLICY IF EXISTS "name" ON table; CREATE POLICY ...` is the safer pattern — it works on PG15 and later and makes intent explicit.
- **`User.email` is `string | undefined`**: The Supabase JS `User` type has `email?: string`. Always guard with `if (!user.email) { throw ... }` before any DB write that requires email. TypeScript narrows after the guard so no further casts are needed.
- **Provisioning in callback route, not DB trigger**: User provisioning (INSERT/UPDATE) belongs in `app/auth/callback/route.ts` (runs on every login), not a Supabase trigger on `auth.users` INSERT (only fires on new account creation, misses pre-existing accounts that logged in before the trigger was deployed). Callback-based provisioning is always safe (idempotent upsert) and handles all users correctly (STORY-03, ADR-004).
- **Conditional upsert pattern for user-editable fields in OAuth provisioning** (STORY-21): When a callback handler upserts user data on every login and some fields are user-editable (e.g., `display_name` set via `/settings`), the UPDATE payload must be conditional — fetch the current value in the same SELECT that checks existence, then only include the field in the UPDATE if the current DB value is empty/default. This prevents silently overwriting a user's self-service edit on every returning login. Pattern: `const { data: existing } = await client.from('users').select('id, display_name').eq('id', userId).maybeSingle(); if (existing) { const updatePayload = { email: ... }; if (existing.display_name === '') updatePayload.display_name = oauthName; await client.from('users').update(updatePayload).eq('id', userId); }`. This is a data-integrity safeguard; the alternative (unconditionally overwriting) is a silent bug that only manifests after the self-service UI ships.
- **`SUPABASE_SERVICE_ROLE_KEY` in CI**: Add `SUPABASE_SERVICE_ROLE_KEY: placeholder` to both Build and Smoke test env blocks in `.github/workflows/ci.yml`, following the same pattern as `NEXT_PUBLIC_*` vars — even before the key is actually used at runtime, to prevent future gaps when service-role operations are added.
- **Bootstrap-first-admin guard**: Count rows in `public.users` (not `auth.users`). After the count query, guard `count === null` with a throw before checking `count === 0`. Do not use `count ?? -1` after a null-throw — it's dead code. Null count means the query itself failed; treat as a fatal error to prevent incorrect admin promotion.
- **SECURITY DEFINER + REVOKE/GRANT for RLS helper functions**: When an RLS policy needs to read from the same table it protects (risking infinite recursion), use a `SECURITY DEFINER` function with `SET search_path = ''`. Immediately after `CREATE OR REPLACE FUNCTION`, add `REVOKE EXECUTE ON FUNCTION fn() FROM PUBLIC; GRANT EXECUTE ON FUNCTION fn() TO authenticated;` — this prevents unauthenticated callers from invoking it directly. See `supabase/migrations/20260628000002_rls_admin_policies.sql`.
- **Supabase Table Editor and SQL Editor bypass RLS**: Both tools connect as the postgres superuser and ignore all RLS policies. Never use them to verify that RLS is working correctly. Instead, test via `fetch('/api/...')` from an authenticated browser session — this goes through the anon-key client and respects RLS.
- **`DROP FUNCTION IF EXISTS` before `CREATE OR REPLACE FUNCTION` is redundant**: `CREATE OR REPLACE FUNCTION` handles the existing-function case idempotently. The preceding DROP is unnecessary and creates a brief functional gap. Use only `CREATE OR REPLACE FUNCTION` in migrations.
